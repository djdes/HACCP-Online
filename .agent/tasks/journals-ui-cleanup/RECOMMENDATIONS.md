## Топ-10 рекомендаций — что добавить/улучшить до завтра и потом

### 🔴 СРОЧНО до завтрашнего интеграционного дня

1. **Глобальная валидация payload'а во внешнем API**. Сейчас если внешнее приложение пришлёт `{quantity: "5 kg"}` вместо `{quantity: 5, unit: "kg"}`, диспетчер сохранит как есть и в UI/PDF появится мусор. Нужен типизированный per-code схематоп (Zod) с понятными ошибками. См. `src/lib/external/dispatch.ts:687` — дописать `validatePayloadForCode(code, data)` в начале.

2. **Идемпотентность `POST /api/external/entries`**. Если внешнее приложение случайно повторит POST через retry — должна сработать апсёрт-логика, а не появиться дубль. Сейчас upsert по `(documentId, employeeId, date)` — это работает только для entry-writer; config-writer (20 журналов) тупо дописывает строки в `JournalDocument.config.rows`. **Это серьёзный риск дубликатов**. Нужно: добавить `Idempotency-Key` header → таблица `JournalExternalIdempotency(key, response)`, при повторе возвращать прошлый ответ.

3. **Rate-limit + IP allow-list** для `EXTERNAL_API_TOKEN`. Сейчас токен один, и любой кто его узнает может писать всё что угодно. Минимум — middleware с rate-limit (упрощённо: in-memory map), желательно — отдельные токены per-organization + IP whitelist в `Organization.allowedIps`.

4. **Аудит-сумма за день** (`/api/external/summary?orgId=X&date=Y`) — endpoint, который возвращает «сегодня заполнено N из 35 журналов», чтобы внешнее приложение могло показывать прогресс пользователям.

### 🟡 Полезные UX-фичи на ближайшую неделю

5. **Шаблоны быстрого заполнения**. Внутри документа кнопка «Заполнить как вчера» — копирует прошлую запись с автоматическим инкрементом даты. Для повара экономит 80% кликов.

6. **Telegram-бот для напоминаний**. Для каждого журнала в `JournalTemplate` указать `cron` (например, climate_control = «10:00, 17:00 каждый день»). В `Organization` есть `telegramChatId` — отдельный воркер шлёт «не заполнен журнал X за сегодня» если в `JournalDocumentEntry` нет строки за день.

7. **«Красные флаги» на главной**. Дашборд показывает только зелёный статус. Добавить виджет «За эту неделю не заполнено N документов: А, Б, В». Сейчас это видно только если зайти в каждый журнал.

8. **Поиск по записям внутри документа**. Поиск, который ты добавил на главной — отлично. Но внутри 30-дневного журнала с 9 сотрудниками найти конкретную запись (например, кто болел 14 апреля) можно только глазами. Нужна строчка фильтра в шапке таблицы.

9. **Экспорт в Excel**. PDF подходит для печати, но проверяющие любят попросить «дайте мне выгрузку за квартал». Один общий endpoint `/api/journal-documents/<id>/xlsx` с использованием `xlsx` пакета (он уже в deps — см. product-writeoff).

10. **Бейдж «нужно подписать» на закрытие документа**. Сейчас «Закончить журнал» — кнопка без визуального напоминания. После закрытия документа в PDF появляется блок «Утверждаю __________» с пустой подписью. Добавить дашборд-плитку «N документов ждут подписи руководителя».

### 🔵 Доделать после интеграции (несрочно)

11. **Универсальный PDF-движок**. Сейчас 35 журналов = 35 отдельных функций `drawXxxPdf` в `document-pdf.ts` (5000+ строк). Шрифт регистрируется в одном месте, но каждый journal сам решает где h1, где table. Это причина того что на пяти журналах заголовки обрезались а на одном вообще шрифт ломался. Нужен один `<JournalPdfPage title rows>` который всегда даёт правильный layout.

12. **Per-template поле `requiresSignature: boolean`**. Многие журналы должны быть подписаны разными людьми (повар + контролёр). Сейчас это размазано по `responsibleTitle` / `responsibleUserId`. Чёткое поле + UI-блок подписи + `signedAt` в `JournalDocumentEntry` дадут юридически валидный лог для проверяющего.

13. **Двухфакторка для роли `manager`**. Управляющий — единственный, кто закрывает журналы. Если у него угонят пароль — можно подделать всю историю. TOTP через `otplib` — два часа работы.

14. **Архив за >1 год должен лежать отдельно**. Сейчас все `JournalDocumentEntry` в одной таблице. Через год при 50 журналах × 30 дней × 20 сотрудников × 12 месяцев = 360k строк только entries. Партиционирование по году + read-only режим для прошлых периодов.

15. **Webhook'и наружу**. Сейчас внешнее приложение пушит в нас. Нужна обратная петля: «на сегодня осталось не заполнено N журналов» → POST на `Organization.webhookUrl` чтобы внешнее приложение могло показать список нотификаций пользователю.

## Будут ли люди этим пользоваться?

**Что хорошо:**
- 35 канонических журналов с сохранением СанПиН-структуры. Привычно для любого технолога.
- PDF на печать выглядит как «настоящий» бумажный журнал — отдел контроля примет.
- Внешний API уже работает — можно интегрировать с любым POS / приложением сотрудника.
- Тарифы Basic/Extended визуально разделены — не пугают «всеми кнопками сразу».
- Print-кнопка наверху каждого документа — закрыли реальную UX-боль (раньше прятали в три точки).

**Что напрягает реального пользователя (повара/уборщика):**
- **Слишком много кликов на одну запись.** Чтобы внести температуру холодильника надо: открыть журналы → найти нужный → открыть документ → кликнуть в ячейку дня → ввести значение → сохранить. Через мобильное приложение сотрудника (которое подключают завтра) этот путь ужмётся до одного нажатия — это правильное направление.
- **Документ за месяц на одной странице** в PDF — некоторые journal'ы выходят на 2-3 страницы с микро-шрифтом. Уборщик / технолог при печати на A4 не сможет прочитать без увеличения. Выход: кнопка «PDF за неделю», по дефолту печатать недельные срезы.
- **Нет автозаполнения «выходной»**. Если сотрудник в пятницу в отпуске, надо вручную помечать каждую ячейку «В». В hygiene/health эта механика есть, в остальных нет.
- **Должность хранится через `position` который не везде используется**. Я зафиксил три места, но есть ещё журналы где `getUserRoleLabel(role)` срабатывает раньше `user.position`. На скриншоте который ты прислал сегодня — был баг с «engineer» как должность. Это user data, но UI должен показать «Инженер» (русский label) или хотя бы предупредить «должность не задана».
- **Нет offline-режима**. Если на кухне отключился wifi — приложение сотрудника должно копить записи локально и слить когда вернётся сеть. Это уже задача внешнего приложения, но наш API должен принимать batch-POST с прошлыми датами.

## Финальный совет — что я бы сделал ПЕРЕД интеграцией завтра

1. **Сгенерировать отдельный токен per-organization**, а не один общий. Сделать `Organization.externalApiToken` (nullable), middleware читает Authorization Bearer и резолвит organization. Это 20 минут работы и убирает класс багов «приложение случайно записало в чужую организацию».

2. **Обернуть `dispatchExternalEntries` в idempotency-check** (см. рекомендацию №2). 30 минут.

3. **Добавить healthcheck `/api/external/healthz`** который возвращает `{ok, build, supportedJournals: 35, dbReachable}` — внешнее приложение должно проверять этот endpoint при старте. 5 минут.

4. **Документировать в `API.md` shape payload'а для каждого из 35 кодов** (какие поля обязательны, какие дефолты). Сейчас в API.md только curl-примеры, контракт неявный. 1 час.

После этого интеграция реально полетит без сюрпризов.
