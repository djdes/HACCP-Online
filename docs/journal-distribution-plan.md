> Один большой проект разбит на три независимые фазы. Делается строго
> по порядку. Этот файл — единственный источник истины «где мы сейчас».

---

## ⚠️ Правила работы по этому плану

1. **Идём сверху вниз внутри фазы.** Не прыгаем.
2. **Пока в текущей фазе остался хоть один незавершённый пункт —
   следующую фазу не начинаем.**
3. **Завершённый пункт удаляется из файла** — не зачёркивается, а
   вырезается целиком (вместе с комментариями к нему). Это коммитится.
4. **Когда фаза становится пустой** (ни одного пункта не осталось) —
   удаляется заголовок фазы и её преамбула.
5. **Когда файл стал пустым** (нет ни одной фазы) — задача целиком
   завершена и файл `docs/journal-distribution-plan.md` удаляется.
6. **Перед стартом фазы**: все её 🔴 открытые вопросы должны иметь
   зафиксированные ответы прямо в этом файле. Если не зафиксированы —
   спросить пользователя и записать ответы перед кодом.
7. **Не открывать другие фичи** мимо этого плана пока он не пуст.
   Багфиксы и UX-полировка — можно (отдельные коммиты).

Чтобы продолжить в новом чате — просто скажи: «читай
docs/journal-distribution-plan.md и продолжай».

## Фаза 1 — Smart Routing (A + B)

> Решает главную боль: менеджер жмёт «отправить всё в TasksFlow», и
> сейчас одна задача уходит ВСЕМ. Цель фазы: каждому шаблону зашит
> режим распределения и список разрешённых должностей; обязательства
> создаются по этой логике, а не «всем подряд».

### 🔴 Открытые вопросы (нужны ответы до кода)

- [ ] **Q1.** При `fillMode = "single"` — кто единственный исполнитель?
  - A) Менеджер выбирает одного назначаемого вручную для шаблона
  - B) Round-robin между всеми с допустимой должностью (равная нагрузка)
  - C) Первый-кто-успел — все видят задачу, но как только один открыл,
     остальным она помечается «уже взято Иваном»
  - D) Гибрид: менеджер либо назначает конкретного, либо «авто» —
     тогда round-robin
  - **Моё предпочтение: D** (даёт менеджеру контроль + автоматику когда
     не хочет морочиться). C опасно — гонка, и если человек открыл и
     ушёл болеть, задача застряла.

- [ ] **Q2.** Где менеджер настраивает `fillMode` и допустимые
  должности?
  - A) Раздел `/settings/journals` (там уже есть toggle включить/выключить)
  - B) Внутри каждого журнала (`/journals/[code]` → новый таб «Настройки»)
  - C) Новая страница `/settings/journal-distribution` (отдельный hub)
  - **Моё предпочтение: A** (рядом со «включить/отключить» — логически
     одно и то же место «как мы ведём этот журнал в нашей орге»).

- [ ] **Q3.** Дефолт `fillMode` для существующих шаблонов на миграции —
  жёстко прошить через `seed.ts`, или дать менеджеру пройтись и выбрать
  при первом заходе?
  - **Моё предпочтение: миграция с разумными дефолтами** (hygiene,
     health, ppe-issuance → per-employee; всё остальное → single).
     Менеджер всегда сможет поправить.

- [ ] **Q4.** «Допустимые должности» для шаблона — это white-list
  должностей в организации, ИЛИ хардкод-роли (`manager`, `cook`,
  `waiter`)?
  - **Моё предпочтение: white-list `JobPosition`** — должности per-org
     (у одного «холодный повар», у другого «бариста»), хардкод-роли
     слишком грубо.

### Шаги

- [ ] **1.1.** Schema: добавить в `JournalTemplate`
  - `fillMode String @default("per-employee")` — enum-like:
    `"single" | "per-employee" | "sensor"`
  - `defaultAssigneeId String?` — для `fillMode="single"` если менеджер
    назначил конкретного (Q1.D), иначе null = round-robin
- [ ] **1.2.** Schema: новая таблица `JobPositionJournalAccess`
  ```
  id, organizationId, jobPositionId, templateId, allowed Boolean
  @@unique([jobPositionId, templateId])
  ```
- [ ] **1.3.** `prisma db push` локально + проверка типов
- [ ] **1.4.** Helper `lib/journal-routing.ts` с функциями:
  - `getEligibleEmployees(orgId, templateId)` — на основе должностей
    + UserJournalAccess override
  - `pickSingleAssignee(orgId, templateId, dateKey)` — round-robin
    или defaultAssigneeId
  - `getFillMode(template)`
- [ ] **1.5.** Переписать `bulk-assign-today` (и аналоги — найти все
  места где создаются `JournalObligation`):
  - Если `fillMode === "per-employee"` — старая логика, обязательство
    каждому из getEligibleEmployees
  - Если `fillMode === "single"` — одно обязательство на pickSingleAssignee
  - Если `fillMode === "sensor"` — не создаём ничего (Фаза 2 заменит)
- [ ] **1.6.** UI в `/settings/journals` (одно место — Q2.A):
  - На каждой карточке шаблона — селект `fillMode` (3 опции с пояснениями)
  - Под ним: чекбоксы должностей орги (Q4)
  - Если `single` — селект «по умолчанию назначать [имя или авто]»
- [ ] **1.7.** Migration script `prisma/seed-journal-fill-modes.ts`:
  - Для всех `JournalTemplate` без `fillMode` — выставить дефолт по
    хардкод-маппингу (Q3): hygiene/health/ppe → per-employee, остальное
    → single
  - Идемпотентно
- [ ] **1.8.** Ручной QA на проде по чек-листу
  - Менеджер выбрал `single` для cleaning — после bulk-assign только
    один сотрудник видит задачу
  - Менеджер выбрал `per-employee` для hygiene — все 5 поваров видят
  - Запретил уборщице фритюр — fryer-oil не приходит уборщице
- [ ] **1.9.** TasksFlow sync — проверить что переписанные obligations
  корректно создают TasksFlow-задачи через адаптеры
