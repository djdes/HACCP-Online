> Один большой проект разбит на три независимые фазы. Делается строго
> по порядку. Этот файл — единственный источник истины «где мы сейчас».

---

## ⚠️ Правила работы по этому плану

1. **Идём сверху вниз внутри фазы.** Не прыгаем.
2. **Пока в текущей фазе остался хоть один незавершённый пункт —
   следующую фазу не начинаем.**
3. **Завершённый пункт удаляется из файла** — не зачёркивается, а
   вырезается целиком (вместе с комментариями к нему). Это коммитится.
4. **Когда фаза становится пустой** (ни одного пункта не осталось) —
   удаляется заголовок фазы и её преамбула.
5. **Когда файл стал пустым** (нет ни одной фазы) — задача целиком
   завершена и файл `docs/journal-distribution-plan.md` удаляется.
6. **Перед стартом фазы**: все её 🔴 открытые вопросы должны иметь
   зафиксированные ответы прямо в этом файле. Если не зафиксированы —
   спросить пользователя и записать ответы перед кодом.
7. **Не открывать другие фичи** мимо этого плана пока он не пуст.
   Багфиксы и UX-полировка — можно (отдельные коммиты).

Чтобы продолжить в новом чате — просто скажи: «читай
docs/journal-distribution-plan.md и продолжай».

## Фаза 2 — Sensors / IoT

> Цель: вычистить из штатных задач то что должен делать датчик
> (температура холодильников, влажность, УФ-лампа), сэкономить смене
> 5–10 минут в день.

### ✅ Зафиксированные решения (2026-04-25)

- **Q5 → manual entry в MVP**: менеджер вручную вводит Tuya `deviceId`
  в `/settings/equipment` (он его видит в приложении Tuya). Auto-discover
  через OAuth — отдельная задача после MVP, не блокирует Фазу 2.
- **Q6 → hybrid failover**: если последнее значение от датчика старше
  24 часов — система создаёт обычное obligation для штатных
  сотрудников (как при `fillMode="per-employee"` из white-list-а
  должностей) + push-уведомление менеджеру «датчик X не отвечает».
  Когда датчик ожил — следующая cron-итерация снова возьмёт
  авто-данные.

### Шаги

- [ ] **2.3.** Worker / cron: когда `fillMode === "sensor"`, вместо
  создания obligation — забирать значения с датчика и авто-создавать
  записи в журнал
- [ ] **2.4.** `/settings/equipment` — UI «привязать датчик к
  шаблону + полю»
- [ ] **2.5.** Failover (Q6 = hybrid): если датчик не отвечает > 24
  часов — создаём obligation для штата + push менеджеру

## Фаза 3 — Bonus + Photo Proof (после Фазы 2)

> Цель: мотивационная подсистема. Сверх-работа (журналы доп. бонусные)
> оплачивается отдельно, премия идёт первому-кто-выполнил, фото-доказ
> обязательно. Превращает заполнение в гонку — поэтому очень рискованно
> делать без хорошего фундамента из Фаз 1+2.

### 🔴 Открытые вопросы (когда дойдём — обсуждать долго)

- [ ] **Q7.** Размер бонуса — фиксированный per-template (как в идее,
  «+50₽ за журнал»), или менеджер ставит сам?
- [ ] **Q8.** Куда списывается бонус? Отдельная статья «премия по
  задачам» в payroll-выгрузке или просто учёт в кабинете?
- [ ] **Q9.** Manager approval после фото — обязательный (медленнее,
  но честнее) или auto-approve если фото есть (быстрее, но fraud-risk)?
- [ ] **Q10.** Что если двое одновременно отправили запись (race
  condition в DB)? Нужен `SELECT FOR UPDATE` + чёткое правило «выиграл
  тот чья транзакция первой коммитнулась», и UI «опоздал».

### Шаги (черновик, уточнится перед фазой)

- [ ] **3.1.** Schema: `BonusEntry { obligationId, amount, status,

- [ ] **3.2.** Per-template setting `bonusAmount` (Q7)
- [ ] **3.3.** API: `POST /api/journals/.../claim-bonus` с
  `SELECT FOR UPDATE` на `JournalObligation` — кто первый, тому
  обязательство и `BonusEntry pending`
- [ ] **3.4.** Photo upload pipeline (предположительно уже есть
  `JournalEntryAttachment` — переиспользовать)
- [ ] **3.5.** Manager-approval UI (Q9)
- [ ] **3.6.** Payroll-выгрузка / отчёт «Премии за период» (Q8)
- [ ] **3.7.** Анти-фрод: для photo проверять EXIF timestamp, deny
